## PHI-013 — Coordinator Trusts Disk, Not Worker Summaries

**Date:** 2026-04-26
**Domain:** architecture
**Source Project:** Claude-Root
**Source:** Spec 010 (autonomous-workflow) review surfaced that having the orchestrator parse subagent-returned summaries as audit truth created an unfixable silent-failure class — a subagent could omit, hallucinate, or lie in its summary and the orchestrator had no independent observation surface to detect it. Inverted the model (ADR-013): subagent writes to decision-log on disk during execution; orchestrator reads disk.

### Philosophy
When a coordinator dispatches workers and parses their returned summaries as audit truth, the coordinator has no independent observation surface — lying, hallucinating, or omitting summaries are an unfixable failure class. Prefer architectures where workers write directly to a shared on-disk artifact during execution; the coordinator reads disk for routing decisions and never trusts a returned string as the source of truth.

### Why I Hold This
The spec-010 review (devils-advocate Phase B) found that the original model — orchestrator synthesizes a log entry from the subagent's returned summary — has no way to catch a subagent that returns "looks fine" while having actually skipped the work, hallucinated success, or omitted a critical decision. The audit trail, which is supposed to be the safety mechanism, becomes a fiction generated by the same actor it is supposed to audit. Disk-as-source-of-truth restores independence: the orchestrator's routing decisions are made on artifacts the orchestrator can read directly, not on a string the subagent chose to return.

### Where It Applies
- Multi-agent LLM orchestration (this case)
- Distributed task runners that route on worker-reported status
- CI/CD pipelines that branch on worker reports rather than worker artifacts
- Any pattern where dispatcher and worker operate in separate contexts and the dispatcher needs to verify the worker actually did what it claims

### Known Tensions
- Disk write contention / locking when many workers run in parallel — needs a write protocol (single-writer-per-stage, append-only with worker ID prefix, etc.)
- Latency: disk round-trip is slower than reading a returned value in memory
- Schema rigidity: a structured artifact contract is harder to evolve than a free-form summary
- Some signals (e.g., model confidence) are not naturally artifact-shaped and may still need to ride along in the return value

### Open to Revision When
- The worker is a deterministic, fully-tested, well-understood unit (e.g., a CI step running a known script) — at that point summary-trust is fine because the failure mode being defended against doesn't exist
- The coordinator has an independent observation channel that catches lying summaries (e.g., file-system events, network telemetry) — at which point the summary becomes a hint, not a truth claim
- The cost of the artifact protocol exceeds the cost of the failure mode it prevents (e.g., toy/internal tools where a wrong route is recoverable in seconds)
